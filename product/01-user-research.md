## 2. 调研背景与目标

### 2.1 调研背景

2024 年以来，生成式 AI 与大模型技术进入爆发式增长期，全球范围内几乎每天都有新的论文、模型、开源项目、产品发布与行业事件产生。然而，承载这些信息的渠道高度分散——GitHub、HackerNews、Reddit、各类技术博客 RSS、国内微信公众号、Telegram 频道等——彼此割裂，且以英文为主。中文 AI 从业者要在有限时间内完成「跨平台采集 → 质量判断 → 翻译理解 → 提炼要点」的全流程，时间成本极高。

AIHotNews 定位为「AI 驱动的多源新闻聚合系统」，通过多智能体协作的 Pipeline 自动完成「采集 → AI 评分筛选 → DeepSeek 大模型生成中文简报 → 自动发布到 GitHub Pages」全链路。为了让产品的信息源选择、评分阈值、简报结构、推送方式更贴合真实用户需求，本次调研在产品完成前三阶段（本地跑通、上线发布、品牌化调优）后、进入第四阶段（邮件订阅）前开展，目的是为后续功能优先级排序提供数据支撑。

### 2.2 调研目标

本次调研聚焦四个核心问题：

1. **谁在用**：识别 AIHotNews 的真实目标用户群体，刻画其人口统计、行为与消费习惯。
2. **痛点在哪**：系统梳理中文 AI 从业者在获取 AI 前沿资讯时面临的具体痛点，并量化其普遍程度与严重程度。
3. **需求是什么**：基于 Kano 模型对用户需求进行分类（基本型 / 期望型 / 兴奋型），明确「必须做」与「可以加分」的边界。
4. **产品该怎么改**：将调研结论转化为可执行的产品建议，指导信息源调优、简报结构优化与第四阶段邮件订阅功能的设计。

## 3. 调研方法

本次调研采用三种方法组合，互相印证，避免单一方法的偏差。

### 3.1 桌面调研（Desk Research）

- **对象**：国内外 AI 资讯聚合类产品（36氪快讯、机器之心、量子位、Hacker News、Reddit r/MachineLearning、Feedly、Inoreader 等）的公开功能介绍、用户评论、应用商店评分。
- **目的**：了解现有产品的信息源覆盖、输出形式、AI 能力与商业模式，识别市场空白。
- **产出**：竞品功能对比矩阵（详见 02-competitor-analysis.md）。

### 3.2 深度访谈模拟（In-depth Interview Simulation）

- **对象**：基于目标人群特征，构建 4 类典型用户画像，针对每类画像设计访谈提纲并模拟深度访谈。
- **方法**：采用半结构化访谈，围绕「每日信息获取流程」「当前工具与不满意点」「对 AI 自动化简报的期待」三条主线展开。
- **样本**：每类画像模拟 2–3 个具体角色实例，合计 10 个角色实例。
- **局限说明**：因资源限制，本阶段以画像推演为主，后续应辅以真实用户访谈验证。

### 3.3 社区舆情分析（Community Sentiment Analysis）

- **对象**：Reddit（r/MachineLearning、r/LocalLLaMA）、V2EX、即刻、知乎、X（Twitter）等社区中关于「AI 资讯获取」「信息过载」「工具推荐」的讨论帖与高赞评论。
- **方法**：关键词检索 + 主题聚类，提取高频抱怨与需求表达。
- **产出**：痛点频次排序与典型用户原话引用。

## 5. 用户痛点深度分析

基于社区舆情与访谈模拟，识别出七类核心痛点，按提及频次与严重程度排序。

### 5.1 信息过载（高频 / 高严重）

**表现**：单日 AI 相关信息量可达数百条，分布在十余个平台，人工筛选耗时 1–2 小时仍难覆盖全面。
**根因**：信息生产端去中心化，无统一聚合与质量分级。
**典型原话**：「每天光刷 Twitter 和 HN 就要一个小时，还总怕漏掉重要的。」

### 5.2 语言障碍（中高频 / 中高严重）

**表现**：一手信息以英文为主，国内从业者阅读速度受限；机翻质量参差，专业术语翻译不统一。
**根因**：中文高质量 AI 资讯供给不足，依赖人工翻译存在时滞。
**影响**：技术管理者与学生群体尤其受影响，前者时间紧，后者语言能力有限。

### 5.3 质量参差（高频 / 高严重）

**表现**：不同信源质量差异大；部分公众号标题党、夸大表述；Reddit/Telegram 噪声多。
**根因**：缺乏统一的质量评估与筛选机制。
**用户期望**：有可信的「重要性评分」帮助快速决策是否深读。

### 5.4 时效性不足（中频 / 中严重）

**表现**：国内媒体转译海外信息存在 6–24 小时时滞；RSS 更新频率不一。
**根因**：人工编辑流程慢；自动化程度低。
**对产品启示**：每日定时运行 + 自动发布可显著缩短时滞。

### 5.5 信源管理困难（中频 / 中严重）

**表现**：用户需自行维护多个 RSS 订阅、多个 App、多个群组；新增源与剔除失效源成本高。
**根因**：工具分散，缺乏统一配置入口。
**用户期望**：开箱即用的预设信源集合，并支持自定义增删。

### 5.6 缺乏深度解读（中频 / 中严重）

**表现**：多数聚合工具只做标题罗列，缺乏「为什么重要」「对谁有影响」的背景解读。
**根因**：人工解读成本高；多数工具未接入大模型做内容增强。
**对产品启示**：enricher 模块的背景补充是差异化价值点。

### 5.7 中英文信息鸿沟（中低频 / 中严重）

**表现**：国内读者与海外社区存在认知时差；同一事件中英文报道角度不同，缺乏对照。
**根因**：信源割裂，缺少跨语言整合。
**用户期望**：同一份简报同时呈现国内外视角。

## 6. 用户需求优先级矩阵（Kano 模型）

将上述痛点与用户期望转化为具体需求，按 Kano 模型分类。

### 6.1 基本型需求（Must-be）——不满足则用户流失

| 需求 | 说明 | 对应痛点 |
|------|------|----------|
| 多源聚合 | 至少覆盖 GitHub/HN/Reddit/RSS/国内媒体等主流渠道 | 信息过载、信源管理困难 |
| 中文输出 | 简报全中文，术语统一 | 语言障碍 |
| 质量筛选 | 有评分机制过滤低质量内容 | 质量参差 |
| 每日定时产出 | 稳定的更新频率 | 时效性不足 |
| 可访问的发布站点 | GitHub Pages 稳定可访问 | — |

### 6.2 期望型需求（One-dimensional）——做得越好满意度越高

| 需求 | 说明 | 对应痛点 |
|------|------|----------|
| 重要资讯排序 | 按评分排序，突出 Top 条目 | 质量参差 |
| 背景解读增强 | enricher 补充「为什么重要」 | 缺乏深度解读 |
| 中英文源并覆盖 | 同一简报含国内外视角 | 中英文信息鸿沟 |
| 原始链接可追溯 | 每条附参考链接 | 质量参差 |
| 飞书卡片推送 | 生成后自动推送到群 | 时效性、信源管理 |
| 自定义信源 | 支持增删 RSS 源 | 信源管理困难 |

### 6.3 兴奋型需求（Attractive）——超出预期，提升忠诚度

| 需求 | 说明 | 对应画像 |
|------|------|----------|
| 邮件订阅 | 每日简报自动送达邮箱，便于归档 | 技术管理者、投资人 |
| MCP 协议接入 | 允许第三方 Agent 调用采集与分析能力 | 开发者 |
| 分阶段 Pipeline 执行 | 支持 staged 执行与运行产物查询 | 开发者 |
| 周报/月报聚合 | 周期性回溯重要趋势 | 投资人、管理者 |
| 个性化评分偏好 | 用户可调整重要资讯阈值 | 高级用户 |

## 7. 调研结论与产品建议

### 7.1 核心结论

1. **目标用户收敛**：四类画像中，AI 开发者与技术管理者是当前最高价值群体，前者贡献技术影响力与口碑，后者贡献付费意愿与传播。学生群体是长期增长池，但短期付费能力弱。
2. **痛点优先级**：信息过载与质量参差是最高优先级痛点，直接决定用户是否留存；语言障碍与缺乏深度解读是核心差异化机会点。
3. **需求边界清晰**：基本型需求已由前三阶段满足（多源聚合、中文输出、评分筛选、定时发布）；期望型需求中「背景解读增强」与「中英文并覆盖」是当前应重点打磨的方向；兴奋型需求中「邮件订阅」是第四阶段的明确目标。

### 7.2 产品建议

1. **信息源调优**：保持 12 类源的广度优势，重点保证国内源（量子位、新智元、36氪、机器之心、虎嗅）的稳定性与国内 RSS 源的覆盖，解决社区反馈的「国内源偶有失效」问题。
2. **评分阈值可配置化**：当前重要资讯阈值固定为 8.0，建议在配置层面暴露该参数，满足不同用户对「严格筛选 vs 宽松覆盖」的偏好。
3. **简报结构优化**：在现有「排名列表 + 分条详情」基础上，强化 enricher 产出的背景解读，明确标注「为什么重要」「影响谁」，呼应技术管理者与学生的诉求。
4. **加速邮件订阅落地**：作为第四阶段目标与兴奋型需求，邮件订阅能显著提升管理者与投资人画像的留存。建议优先实现「每日简报 HTML 邮件推送 + 订阅/退订入口」，归档与周报可作为后续迭代。
5. **MCP 能力对外传播**：MCP 协议接入是面向开发者画像的强差异化能力，建议补充文档与示例，便于第三方 Agent 集成，扩大生态影响力。
6. **建立真实用户反馈通道**：当前调研以画像推演为主，建议在 GitHub Pages 站点加入反馈入口（Issue 模板或表单），持续收集真实用户评价以校准画像。
